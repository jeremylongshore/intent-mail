================================================================================
                          TECHNICAL SPECIFICATION
================================================================================

Message Protocol:
  - Transport: HTTP/HTTPS with JSON-RPC 2.0
  - Alternative: gRPC, HTTP/REST, Server-Sent Events (SSE)
  - Discovery: AgentCard at /.well-known/agent-card.json
  - Operations: send_message, stream_message, get_task, cancel_task

Core Data Structures:
  1. AgentCard (Discovery)
     - Metadata: name, version, description
     - Capabilities: streaming, push notifications
     - Skills: discrete functions agent can perform
     - Security: OAuth2, API keys, mTLS requirements

  2. Message (Communication)
     - Parts: TextPart, FilePart, DataPart
     - Multimodal: supports any MIME type
     - Context: maintains conversation continuity

  3. Task (Work Unit)
     - Lifecycle: 9 states (submitted, working, completed, failed, etc.)
     - Artifacts: output generated by agent
     - Status: includes state, message, timestamp
     - Context: preserves conversation across interactions

  4. Session (State Management)
     - Events: conversation history
     - State: agent's working memory
     - Memory: long-term storage across sessions
     - Automatic: managed by Session Service

Error Handling:
  - JSON-RPC error codes with custom semantics
  - Task states for failures (failed, rejected, cancelled)
  - Recovery patterns: retry, fallback, timeout
  - Resilience: circuit breaker, exponential backoff

                          IMPLEMENTATION PATTERNS
================================================================================

1. RECOMMENDED PATTERN: Multi-Agent with Root Orchestrator
   Root Agent (ADK)
     ├─ Email Parser Agent (A2A)
     ├─ Intent Classifier Agent (A2A)
     └─ Response Generator Agent (A2A)

   Benefits:
   - Clear separation of concerns
   - Independent agent scaling
   - Shared session state
   - Easy to add new agents

2. SESSION MANAGEMENT
   - Automatic session creation on first interaction
   - State: agent's short-term working memory (dict)
   - Memory: long-term storage across sessions
   - Callbacks: before_agent_run, after_tool_run

3. TASK DELEGATION
   - Discover agent capabilities via AgentCard
   - Select specialized agent by skill
   - Send message with context preservation
   - Poll or stream results back

4. ERROR RECOVERY
   - Retry with exponential backoff
   - Circuit breaker for cascading failures
   - Timeout management for long-running tasks
   - Fallback to alternative agents if available

================================================================================
                          COMPARISON WITH ALTERNATIVES
================================================================================

A2A vs. MCP (Model Context Protocol):
  A2A: Agent-to-Agent communication (horizontal)
  MCP: Agent-to-Tool communication (vertical)
  → Complementary, use both together

A2A vs. LangGraph:
  A2A: Cross-framework, vendor-neutral
  LangGraph: Framework-specific, faster for single framework
  → A2A better for multi-framework, multi-vendor scenarios

A2A vs. Microsoft AutoGen:
  A2A: Open standard, conversation-based
  AutoGen: Enterprise-grade conversation framework
  → A2A more standardized, AutoGen more feature-rich

A2A vs. CrewAI:
  A2A: Interoperability protocol
  CrewAI: Feature-rich agent framework
  → A2A for orchestration, CrewAI for individual agents

Verdict: A2A is best for Intent Mail because:
- Multi-agent architecture is core requirement
- Need vendor-neutral platform independence
- Want production-ready with monitoring
- Session management is critical
